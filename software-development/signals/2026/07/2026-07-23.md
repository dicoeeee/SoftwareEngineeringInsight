### 信号

Cursor 发布面向 Teams 与 Enterprise 的 Cursor Router。它在模型执行前按 Query、Context、Task Complexity 与 Domain 对每次请求分类，再在允许的模型池中选择执行模型；训练数据来自 60 万条以上真实请求，线上 A/B Test 覆盖数百万请求，并同时观察用户满意度代理指标、生成代码 Keep Rate 与因切换模型造成的 Cache Miss 成本。产品提供 Intelligence、Balance、Cost 三种优化模式，管理员可以按团队设置模式并允许或阻止具体模型。Cursor 披露的 30%–60% 成本节省、满意度与 Cost per Commit 均属于厂商基于自身生产流量的主张。

### 为什么重要

模型选择开始从用户为整个会话指定一个“日常主模型”，转向 Harness 对每次请求执行的在线决策。可复用的机制不是某个模型排名，而是把路由输入、允许模型集合、缓存成本、用户纠错信号和代码留存共同纳入评估；这也意味着路由策略本身需要版本、审计、回退和数据治理。

### 尚待确认

- 60 万训练请求与数百万 A/B Test 请求的抽样、分组、统计方差和任务复杂度分布。
- 满意度分类器与 Keep Rate 对真实任务成功、代码质量和长期维护成本的代表性。
- 路由策略变更、模型下线、隐私模式和团队模型禁用规则发生冲突时的失败与回退语义。

### 信号

OpenAI Codex 已合并 PR #34852，修复 Durable Sleep 与 Agent Mailbox 之间的唤醒缺口：此前仅排队、不触发新 Turn 的 Agent Mail 不会恢复已进入持久睡眠的空闲 Root Thread，消息虽然持久化，任务却可能一直停滞。修复把 Pending Mailbox Message 视为 Durable Sleep 的待处理工作并触发唤醒，同时保留普通空闲线程仍须显式 `trigger_turn` 的约束；新增回归测试覆盖睡眠 Root Thread 被 Queue-only Mail 唤醒并保留消息。

### 为什么重要

持久睡眠把线程的运行态与进程生命周期解耦后，消息队列就必须参与唤醒判定。可复用的设计是区分“允许唤醒的持久等待状态”和“仍需显式触发的普通空闲状态”，并让消息持久化、线程恢复与消费确认共享同一不变量；否则系统可能出现消息已送达但永不处理的静默停滞。

### 尚待确认

- 多条并发消息、重复投递、进程崩溃和恢复竞态下是否保持一次唤醒与至少一次处理语义。
- 子线程、非 Root Thread 与跨设备恢复是否使用相同 Mailbox 判定。
- 修复进入稳定 Release 的时间，以及实际生产中受影响线程数量与停滞时长。
