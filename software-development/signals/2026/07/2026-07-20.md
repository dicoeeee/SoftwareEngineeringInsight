### 信号

Simon Willison 在自己的 macOS arm64 机器上检查 Claude Code 二进制，`strings` 输出包含 `Bun v1.4.0 (macOS arm64)` 及 563 条 `.rs` 源路径，构成 Claude Code 分发件已包含 Bun Rust Port 的独立实机证据。其文章同时转述 Bun 团队说法：Claude Code `v2.1.181+` 已采用这一 Runtime，Linux 启动速度约提升 10%；版本范围、跨平台覆盖与性能数字仍属于厂商主张，本次未独立复现。

### 为什么重要

分析判断：Coding Agent 的 Harness Runtime 不是不可见的实现细节。它影响启动延迟、内存与崩溃行为、单文件分发、跨平台一致性以及供应链审计。此次二进制级检查说明，外部观察者可以从实际发布制品验证 Runtime 迁移是否落地；但单机证据不能推断所有版本与平台，也不能替代性能基准。

### 尚待确认

- Claude Code 哪些版本、操作系统与架构已经采用 Bun Rust Port，以及是否存在灰度或回退路径。
- Rust Port 在 Claude Code 中承担的确切 Runtime 边界，和 TypeScript/Native 组件的实际组合。
- 启动速度、内存、稳定性与兼容性变化能否由跨平台独立基准复现。

### 信号

Cline SDK `v0.0.65` 与 CLI `v3.0.45` 发布 Team Runs。同期两项已合并修复揭示其控制面缺口：Teammate 的工具注册表曾暴露仅 Orchestrator 可用的 `team_spawn_teammate`，尽管运行时会拒绝该调用；一份 2026-07-15 事件分析记录 7 个 Teammate 各尝试两次，共浪费 14 个推理轮次，修复改为从 Teammate 工具表移除该能力。另一项修复确认，Teammate 模型流返回 `finishReason: error` 时并未抛异常，Orchestrator 因而把任务标记为 Completed，隐藏错误并绕过 Retry；修复后会写入 Failed、发出 `RunFailed`，并在配置时按 Backoff 重试。

### 为什么重要

分析判断：Multi-agent Team 的能力边界必须在模型可见的 Tool Registry、运行时授权和编排状态机中保持一致。只靠执行时拒绝一个不应出现的工具，会消耗上下文与推理预算；只识别异常而忽略流式协议中的错误结束原因，则会把失败伪装成成功，污染后续依赖任务。角色级最小工具集与统一终止语义应成为 Agent Harness 的可测试契约。

### 尚待确认

- Team Runs 在真实仓库中的失败率、重试上限、幂等性和重复副作用控制。
- 其他角色是否仍存在“模型可见但运行时禁止”的工具，或遗漏的协议级错误终止路径。
- 任务依赖图如何处理部分完成、重试耗尽、Orchestrator 重启与人工接管。
