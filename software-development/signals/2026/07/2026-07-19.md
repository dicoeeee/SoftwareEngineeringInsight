### 信号

Qwen Code `v0.19.12` 将符合条件的顶层一次性 Subagent 在省略 `run_in_background` 时默认改为后台执行，只有显式设置 `run_in_background:false` 才同步返回；嵌套调用和调用方持有 `working_dir` 的调用仍默认前台，带 `working_dir` 的后台启动会被拒绝。对应已合并 PR 同时收紧父 Agent 的委派规则：只分派边界清晰、互相独立且写入范围不重叠的工作，在集成前验证子 Agent 的事实和代码主张；Explore Agent 移除有状态工具，Fork Worker 不再默认提交。另一项已合并 PR 增加 `agents.maxParallelAgentsByModel`，在全局并发上限之下按具体模型限制后台 Subagent，达到上限后让同模型任务排队而不阻塞其他模型。`v0.20.1` 又新增 `fork_turns`，允许派生 Subagent 时选择完整历史、最近若干 Turn 或零历史，把上下文继承范围变成显式控制。

### 为什么重要

多 Agent Harness 的关键问题正从“能否并行”转向“默认如何并行、谁能写哪里、子结果是否被复核，以及不同模型能占用多少资源”。把这些边界同时放进调度器、Tool Surface 和模型指导，可以降低重复工作、脏工作区污染、未验证结论直接进入主流程及单一模型限流拖垮全局队列的风险。分析判断：多 Agent 拓扑开始成为需要显式预算、所有权和验证协议的运行时对象，而不只是 Prompt 中的一句“请并行”。

### 信号

Qwen Code `v0.19.12` 为 Channels 中的持久记忆建立了更明确的修改边界。自然语言触发的记忆更新或删除在落盘前必须展示目标和变化，并等待与动作匹配的显式确认；提案按发送者与目标隔离、60 秒过期、执行前先消费，并使用原始文本做 Compare-and-swap，避免过期或并发确认覆盖新内容。明确的多事实请求可一次提取最多 10 条独立事实并原子保存，精确重复会跳过；新增或替换内容若命中高置信度凭据模式会在写入前拒绝，且不在响应、异常或日志中回显匹配值。稳定版还加入按消息召回相关记忆，并给频道创建的 Session 写入来源实例标识。

### 为什么重要

Agent Memory 不是普通聊天历史：错误写入会持续影响后续规划、工具选择和团队协作。该实现把记忆修改从“模型解释一句自然语言后直接写入”推进到带主体隔离、有效期、单次消费、并发保护和敏感信息门禁的事务过程。分析判断：长期工程上下文开始需要与代码变更类似的身份、审批、原子性和审计语义，而不能只依赖模型判断“用户大概想记住什么”。

### 尚待确认

- 凭据检测采用保守高置信度规则，不覆盖所有 Secret 格式；误报、漏报和组织自定义规则尚未披露。
- 按消息召回的相关性、冲突处理、过期策略、记忆容量和对后续任务质量的影响。

### 信号

OpenCode `v1.18.2` 增加 `subagent_depth`，默认有效上限为 1；当任务分派到达配置的嵌套深度后，运行时拒绝继续启动 Subagent，只有显式提高配置才允许更深递归。对应已合并 PR 报告相关 Tool 测试及 Core、JavaScript SDK Typecheck 通过；完整 OpenCode Typecheck 因宿主争用超时，不能据此视为整个项目验证完整通过。

### 为什么重要

Subagent 能继续创建 Subagent 时，任务树可能在模型驱动下无界扩张，带来成本、上下文、并发、写入冲突和终止语义难以预测的问题。默认把拓扑限制为主 Agent 加一层子 Agent，并要求更深递归显式 Opt-in，是一种简单、可审计且与模型无关的运行时护栏。分析判断：多 Agent 安全边界开始从单次 Spawn 上限扩展到拓扑形状本身。

### 尚待确认

- 深度上限在 Fork、Resume、跨 Session、不同自定义 Agent 与并行任务组合中的计数语义。
- 达到上限时的错误是否能被父 Agent 可靠理解并降级执行，而不是反复重试。
- 默认一层对复杂任务成功率、成本和真实递归失控事件的影响。
