### 信号

GitHub 在 Visual Studio 的 2026 年 6 月更新中加入 MCP Server 信任基线：IDE 启动时比较 MCP Server configuration 与 asset fingerprint；一旦相对已信任基线发生变化，默认要求用户 Review/Approve 后才运行。该产品行为可由官方 Changelog 确认，但它能否覆盖所有配置来源、间接资产和运行期变化仍未公开验证。

### 为什么重要

分析判断：外部工具不应在首次授权后获得永久信任。把配置与资产指纹纳入审批键，可在 Server binary、启动参数或相关资产改变时让旧授权失效；这使 Harness 的信任对象从“Server 名称”推进到“可复核的具体版本状态”，也为 MCP 供应链和配置漂移提供了可落地的控制点。

### 尚待确认

- Fingerprint 覆盖哪些文件、依赖、环境变量与远程资源，是否存在只改间接资产而不触发审批的路径。
- 多 Workspace、同步配置和组织级策略下，信任基线的作用域、审计记录与撤销方式。
- Server 在获批后下载或替换资产时，是否存在运行期再验证。

### 信号

Qwen Code 已合并 PR #7005，并在 `v0.19.11-nightly.20260717.f8e6e8931` 中发布 Multi-workspace Ownership Guards：Branch/Fork/`cd` 等 live-session mutation 被明确限制为 Primary Workspace，Secondary 请求以 HTTP 400 失败，且两个 Runtime 均不得执行；REST/ACP 请求继续携带所属 Workspace Environment，测试覆盖 Primary/Secondary 使用同名 Proxy Environment Variable 时互不污染。实现还把对 `process.env` 的 Whole-file Exception 收紧为 AST 级 Key/Expression Allowance。`v0.19.12` 已把这项修复带入稳定版，因此“稳定版是否包含该控制”的待确认项已经消除。2026-07-19 合并的 PR #7221 又为 Web Shell 增加可选 Worktree Session：Daemon 在 `<repoRoot>/.qwen/worktrees/<slug>` 创建 Git Worktree，通过 `changeSessionCwd` 把会话切入隔离目录，并把 Worktree 元数据写入 Sidecar，SDK 和 Git Status 查询都携带该目录。该 PR 曾明确披露，Daemon 重启后仅 Sidecar 标识与模型提示能够恢复，有效 CWD 会退回主 Workspace；`v0.20.1` 已发布 Session Load/Resume 时恢复 Worktree Isolation 的实现，关闭这一已知缺口。Worktree 自动清理、Merge-back 和 Worktree 范围 Diff 仍未实现。

### 为什么重要

分析判断：多工作区 Agent 的隔离不能只依赖目录或 Worktree。只要多个 Runtime 共用 Daemon、Process Environment、REST/ACP Route 或 Session Mutation，Harness 就必须为每条可变路由声明所有者，把环境绑定到所属 Workspace，并在所有权不清时 Fail Closed；否则 Secondary Workspace 可能无声使用 Primary 的代理、凭据或会话状态。Worktree Session 把并行写入隔离下沉到真实文件系统副本，但重启后 CWD 与可见标识分离也说明：隔离状态必须作为可恢复运行时状态重新施加，不能只持久化 UI Metadata 或给模型一段提示。

### 尚待确认

- Windows/Linux 上的路由、环境和路径隔离是否与 macOS 一致。
- Branch/Fork/`cd` 的真实所有权模型；当前 Primary-only Contract 是否只是过渡实现。
- Proxy 之外的 Credential、Path、Config、MCP 与子进程环境是否都有端到端 Workspace Boundary Test。
- Worktree Session 的自动清理、Merge-back、Worktree Diff 与 Windows/Linux 验证何时补齐。
