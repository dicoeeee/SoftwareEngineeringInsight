### 信号

Simon Willison 发布的公开访谈中，Claude Code 团队成员称，协作 Agent Claude Tag 会主动监控 Slack 中的问题，以每个频道一份 Markdown 维护共享记忆，由具体 Session 向频道主记忆回写，并提交工程变更；团队称其目前承担约 65% 的 Claude Code 产品工程 PR。团队还描述了渐进式放行：关键区域保留人工 Code Owner，外围代码在约六个月信任积累后才允许自动评审独立放行；导致事故的 PR 会被加入评测集，新模型须同时运行团队与产品评测。上述工作流与比例均属于团队自述，不能等同于独立验证的生产效果。

### 为什么重要

该实践把协作 Agent 的长期上下文、任务发现、变更生成、信任升级和失败回灌组织为一个可持续控制循环，而不是把 Slack 仅当作新的聊天入口。可复用的工程要点是：把记忆限定在明确协作作用域内，让高风险代码始终保留人工所有者，并把真实事故样本转为后续模型与 Harness 的回归门禁；但具体自动化比例不应直接外推到其他代码库。

### 尚待确认

- 65% PR 占比的统计口径、时间窗口、任务复杂度、人工返工量和最终合并率。
- 频道共享记忆的写入审批、冲突处理、过期、敏感信息防护和访问控制。
- 外围代码自动放行所使用的评审门槛、回滚条件，以及事故评测集的覆盖与泄漏控制。
