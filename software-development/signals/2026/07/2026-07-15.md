### 信号

Anthropic 在 Claude Code `v2.1.210` 的官方 Release 中确认并修复一项缺陷：配置为 `isolation: 'worktree'` 的 Subagent 此前仍可能对主仓库 checkout 执行会改变 Git 状态的命令，而不是只在自己的隔离 Worktree 中操作。同一版本还修复了 `.claude/*` 符号链接未及时进入 Sandbox 禁写清单的问题，并加强了 Agent tool 对间接 Prompt Injection 的防护；这些是官方确认的产品修复，不等同于已公开的安全事件或漏洞通告。

### 为什么重要

这说明“Worktree 隔离”不能只作为任务调度标签或目录约定，必须落实到命令目标、仓库路径和写权限的运行时强制检查。对多 Agent 开发系统而言，隔离保证需要用越界 Git 操作、符号链接和不可信内容注入等对抗性用例持续验证，否则并行执行会把单 Agent 的误操作放大为主仓库污染风险。

### 尚待确认

- 受影响版本范围、触发条件、现实暴露频率及是否存在已知主仓库误修改案例。
- 修复后对 Git 子命令、符号链接、嵌套仓库和跨平台路径的完整覆盖范围。
- 间接 Prompt Injection 加固采用的具体检测或隔离机制及其误报、漏报表现。

### 信号

Armin Ronacher 提出一个分析判断：Coding Agent 显著提高个人修改代码库的能力，但大型项目仍受制于团队能否共享系统概念、边界、不变量、所有权和设计缘由。多人分别让 Agent 完成局部合理的变更时，即使代码能够编译、测试能够通过、解释能够即时生成，成员也可能绕过原本通过阅读、提问、Code Review 和跨团队协调形成共同系统模型的过程；因此局部正确并不能证明团队仍具备一致的架构语言。

### 为什么重要

既有 Agent-ready 实践主要强调代码库可导航性、确定性反馈和自动化验证。这一观点补充了另一个不能完全由测试覆盖的风险：局部行为持续正确的同时，人类对系统整体的共享理解仍可能退化。对应的工程应对不是恢复所有低效摩擦，而是把领域词汇、边界、不变量、所有权和关键架构决策显式化，并让跨模块变更接受能够检查整体一致性的评审。

### 尚待确认

- 是否存在可观测指标，能够区分正常的 Agent 提效与共享架构模型正在退化的代码库。
- 架构测试、领域模型、ADR、代码所有权和跨模块 Review 中哪些机制能有效降低该风险。
- 这一现象在不同团队规模、代码库耦合度和 Agent 使用强度下的发生频率与实际成本。

### 信号

Cognition 称其在 FrontierCode 1.1 上运行了 3,000 个评测 Session，对比 Fable 5 与 Opus 4.8 分别单独运行及搭配同一低成本 Sidekick 的四种配置。在该厂商评测中，Fable + Sidekick 的平均得分为 60.7、成本为每次 1.86 美元，Opus + Sidekick 为 54.6 和 2.04 美元；前者 Lead 平均 11.5 Turn、累计输入 545k Token，后者为 26.5 Turn 和 1,679k Token。Cognition 将差异归因于委派时机和交接方式：两者平均都约有三次 Handoff，但 Fable 更早把实现、测试和 Lint 闭环交给 Sidekick，并用约束与完成定义组织 Brief，而 Opus 更常先自行探索和实现再晚期委派。

### 为什么重要

这项厂商研究提示，多 Agent 系统的经济性不能只按模型单价或委派次数估算；Lead Agent 自己保留多少回合、携带多少上下文，以及能否写出可执行且约束完整的 Handoff，可能更接近总成本与质量的关键变量。对 Harness 设计而言，评测指标应同时记录 Lead Turn、上下文累积、委派时机、返工方式和端到端结果，而不是只看 Token 单价。

### 尚待确认

- FrontierCode 1.1 的任务选择、评分方差、Sidekick 型号与配置，以及结果能否由第三方复现。
- 该结论在短任务、串行根因调试、强耦合重构和私有企业代码库中是否仍成立。
- 生产流量中的路由开销、失败重试、人工介入与质量成本是否会改变评测中的成本排序。
