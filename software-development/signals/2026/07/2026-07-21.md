### 信号

OpenHands 在 Agent Canvas `v1.5.2` 修复云会话 READY 事件重复发送。已合并 PR #1854 披露，一次 staging 会话曾对同一任务发出 6 个 `cloud_conversation_ready` 事件；根因是 Route 和多个子组件分别挂载 `useTaskPolling`，每个 Hook 实例只在本地去重，因此共享 React Query 缓存并未消除遥测、导航、元数据和附件消息交接等重复副作用。修复将 `useTaskPolling` 收敛为被动状态消费者，并由 Route-owned Controller（路由级单一控制器）唯一持有一次性副作用。

### 为什么重要

分析判断：共享状态不自动等于 Exactly-once（只发生一次）副作用。Agent Harness 若有多个 UI 或后台消费者观察同一任务状态，应把查询读取与外部副作用分离，并明确唯一所有者，否则重复导航、重复消息和重复遥测会污染真实控制流与审计数据。

### 尚待确认

- 跨标签页、刷新、网络重连和进程恢复后，READY 副作用是否仍保持幂等。
- 其他终态事件是否采用相同单一所有者模式，以及事件消费是否存在持久幂等键。

### 信号

Qoder 1.16.0 将安全检查分为三个层级：对当前任务生成代码做静态模式匹配、对变更做增量语义扫描、对项目做跨文件和跨函数数据流分析，并把触发点放在代码生成、对话和推送环节。Kiro 与 Snyk 的联合实践则通过 Snyk MCP Server 把代码和依赖扫描接入 Agent 会话，以 Agent Hooks 在文件创建或保存时后台触发，并组合 AIBOM 与 Toxic Flow Analysis 检查 AI 组件来源、提示注入和危险数据流。两者都把带文件定位和修复上下文的发现返回开发循环，但公开效果仍是厂商主张。

### 信号

Google Developers Blog 提出将大型 Agent Prompt 视为构建资产：团队维护模块化 Skill 文件和模板，通过 include、变量与宏转译为确定性成品；构建阶段检查缺失导入、未定义变量、循环依赖和依赖图，并在 CI 中用 Golden File 检测成品漂移。运行时以 Progressive Disclosure（渐进披露）只加载任务所需技能；Agent 可以提出 Skill 变更，但须形成受人工评审和评测约束的变更提案，而不是直接修改运行中的控制指令。

### 为什么重要

当 Prompt 承载工具规则、角色约束和团队知识时，手工拼接长文本会让依赖、版本和行为变化难以审计。把 Prompt 分成源码、构建与运行时加载三层，可把原本隐性的指令漂移转化为可在提交前发现的构建失败或差异；同时也为 Agent 自我改进建立“提出变更而非直接改写控制面”的安全边界。

### 尚待确认

- 该方法在大型真实仓库中的构建时延、维护成本、任务成功率和上下文节省尚无公开量化结果。
- 不同模型与 Harness 对同一确定性 Prompt 成品的行为是否稳定，以及宏和条件展开是否引入新的调试难度。
- Golden File 只能发现文本变化，仍需任务级 Eval 判断语义与效果是否退化。
