# 2026-07-18 专题简报

## 今日判断

开发 Agent 的控制边界正从单次 Tool Approval 扩展到“上下文切换后的持续不变量”：MCP 资产变化要让旧信任失效，审批与取消不能被 Compaction 或长执行削弱，多工作区请求和环境变量必须绑定到明确所有者。与此同时，Global Hooks、Slack Multi-repo Environment 与 Portable Automation 表明，团队级规则和工作流正跨 Workspace、协作入口与部署环境迁移，新的风险也集中在作用域、冲突和审计上。

## 核心信号

### `sd-20260714-copilot-mcp-trust-baseline` — Copilot 以 MCP 配置与资产指纹变化触发重新审批

- 主分类：编码与代码修改
- 横向标签：MCP、Trust Baseline、Asset Fingerprint、变更检测、重新审批、IDE
- 自主程度：L2
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：Visual Studio 启动时比较 MCP Server configuration 与已信任 asset fingerprint；检测到变化后，默认要求用户 Review/Approve 才运行。
- 为什么重要：对外部工具的授权不再只绑定 Server 名称；配置或资产发生变化时旧授权失效，为 MCP 供应链和配置漂移提供了具体控制点。
- 对既有认知的影响：从“某次 Tool Call 是否审批”推进到“被授权的工具身份由哪些可复核资产共同定义”。
- 尚待确认：Fingerprint 覆盖面、运行期资产替换、多 Workspace/组织策略下的作用域与撤销。
- 支持来源：[GitHub Copilot in Visual Studio — June update](https://github.blog/changelog/2026-07-14-github-copilot-in-visual-studio-june-update/)

### `sd-20260714-goose-execution-guardrails` — goose 将审批、执行时限与更新来源校验设为运行时护栏

- 主分类：编码与代码修改
- 横向标签：Approval、Context Compaction、Timeout、Cancellation、Fail Closed、Delegate Isolation
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：goose `v1.43.0` 在 Approval 期间禁用 Compaction，为 Shell/Code Mode 设置 Timeout 与 Cancellation，更新来源无法验证时 Fail Closed，并隔离 Delegate 参数和 Code Mode 可见工具。
- 为什么重要：审批、执行、委派和自更新需要共享不可降级的状态机约束，不能把 Permission Prompt 当成唯一安全边界。
- 对既有认知的影响：延续 Claude 授权完整性与后台生命周期信号，但新增了 Compaction、Delegate Session 和更新来源验证三个失效面。
- 尚待确认：默认值、崩溃恢复和并发行为、跨平台一致性、更新信任根与回滚策略。
- 支持来源：[goose v1.43.0](https://github.com/aaif-goose/goose/releases/tag/v1.43.0)

### `sd-20260716-qwen-multi-workspace-ownership` — Qwen Code 加固多工作区会话所有权与环境隔离

- 主分类：开发协作与工程知识上下文
- 横向标签：Multi-workspace、Session Ownership、Environment Isolation、ACP、REST、Fail Closed
- 自主程度：L3
- 证据状态：早期信号
- 来源优先级：P1
- 发生了什么：Qwen Code Nightly 将 Branch/Fork/`cd` 等可变 Session Route 限制为 Primary Workspace，Secondary 请求以 400 失败且不得由任一 Runtime 执行；REST/ACP 请求携带所属 Workspace Environment，并测试同名 Proxy Environment Variable 不跨 Workspace 污染。
- 为什么重要：目录或 Worktree 隔离不足以覆盖共用 Daemon、Process Environment 和 Session Route；每条可变请求都要绑定所有者，并在不明确时 Fail Closed。
- 对既有认知的影响：为 Qwen 默认 Auto Approval 的 Workspace Trust 问题补上多 Runtime 场景，也把隔离对象从文件路径扩展到路由与进程环境。
- 尚待确认：稳定版时间、Windows/Linux、真实 Ownership Model，以及 Credential/Path/MCP/子进程环境的端到端隔离。
- 支持来源：[Qwen Code Nightly](https://github.com/QwenLM/qwen-code/releases/tag/v0.19.11-nightly.20260717.f8e6e8931)、[fix(serve): Harden multi-workspace ownership guards](https://github.com/QwenLM/qwen-code/pull/7005)
