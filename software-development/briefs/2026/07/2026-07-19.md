# 2026-07-19 软件开发 Agent 专题简报

## 今日判断

多 Agent 开发系统正在从“允许并行”进入“约束拓扑、资源与交接”的阶段：Qwen Code 把顶层一次性 Subagent 默认转为后台，同时要求独立写入范围、验证子结果并支持按模型限流；OpenCode 则默认禁止 Subagent 继续递归派生。另一条变化发生在长期上下文：Qwen 的频道记忆写入开始采用主体隔离、显式确认、原子批处理、并发保护与凭据门禁，说明 Memory 正成为需要事务语义的工程控制面。

## 核心信号

### `sd-20260718-qwen-background-delegation-guardrails` — Qwen Code 将顶层 Subagent 默认转为后台，并加入委派与模型级资源护栏

- 主分类：开发协作与工程知识上下文
- 横向标签：Subagent、Background Execution、Delegation、Concurrency、Model Routing、Dirty Worktree
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：稳定版 `v0.19.12` 让符合条件的顶层一次性 Subagent 在省略参数时默认后台执行；已合并 PR 同时要求委派边界独立、写入范围不重叠、子结果先验证再集成，并加入按具体模型的后台并发上限。
- 为什么重要：Harness 开始把调度默认值、写入所有权、结果验证与模型资源预算组织为同一控制面，而不只是鼓励“尽量并行”。
- 对既有认知的影响：强化“异步多 Agent 是基础架构”的基线，同时补充此前缺失的交接协议和模型级容量约束。
- 尚待确认：Windows/Linux 行为、跨 Session 恢复后的模型级限流，以及默认后台对现有 SDK 调用方和真实成本的影响。
- 支持来源：[Release v0.19.12](https://github.com/QwenLM/qwen-code/releases/tag/v0.19.12)、[Delegation defaults and guardrails](https://github.com/QwenLM/qwen-code/pull/7048)、[Per-model sub-agent concurrency limits](https://github.com/QwenLM/qwen-code/pull/6984)

### `sd-20260717-qwen-channel-memory-controls` — Qwen Code 为频道记忆写入增加确认、原子批处理与凭据护栏

- 主分类：开发协作与工程知识上下文
- 横向标签：Memory、Channels、Confirmation、Compare-and-swap、Credential Guard、Atomic Write
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：自然语言记忆更新/删除需要动作对应的显式确认，提案按发送者和目标隔离、60 秒过期、单次消费并以 Compare-and-swap 防止并发覆盖；多事实写入支持最多 10 条原子保存，凭据模式在落盘前被拒绝且不回显。
- 为什么重要：错误的长期记忆会持续影响后续 Agent 行为，必须像代码变更一样具备身份、审批、原子性和敏感信息边界。
- 对既有认知的影响：把“仓库与工程知识成为 Agent 资产”的基线推进到“持久知识写入需要事务控制”。
- 尚待确认：真实消息平台与 Daemon 重启 E2E、凭据检测漏报/误报，以及相关记忆召回的质量和过期策略。
- 支持来源：[Release v0.19.12](https://github.com/QwenLM/qwen-code/releases/tag/v0.19.12)、[Confirm natural memory mutations](https://github.com/QwenLM/qwen-code/pull/7066)、[Save explicit multi-fact memory safely](https://github.com/QwenLM/qwen-code/pull/7092)

### `sd-20260715-opencode-subagent-depth-limit` — OpenCode 默认把 Subagent 递归深度限制为一层

- 主分类：开发协作与工程知识上下文
- 横向标签：Subagent、Recursion、Topology、Depth Limit、Runtime Guardrail
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：OpenCode `v1.18.2` 增加 `subagent_depth`，默认有效上限为 1；达到配置深度后运行时拒绝继续分派，只有显式配置才允许更深递归。
- 为什么重要：这把多 Agent 风险控制从单次 Spawn 数量扩展到任务树形状，避免模型在默认配置下自行构造无界递归拓扑。
- 对既有认知的影响：与 Qwen 的后台委派和按模型限流共同说明，多 Agent Harness 正把拓扑与容量变成显式、可审计的运行时参数。
- 尚待确认：Fork/Resume/跨 Session 的深度计数语义、到达上限后的降级行为及真实任务质量影响。
- 支持来源：[OpenCode v1.18.2](https://github.com/anomalyco/opencode/releases/tag/v1.18.2)、[Limit subagent nesting depth](https://github.com/anomalyco/opencode/pull/37124)
