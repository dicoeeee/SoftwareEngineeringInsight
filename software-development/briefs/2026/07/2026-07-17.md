# 2026-07-17 专题简报

## 今日判断

开发 Agent 的授权边界正在同时向两个方向移动：Qwen Code 把 Workspace 内写入默认交给自动决策，而 Codex 即使在 Full Access 中仍增加独立的破坏性命令拦截。与此同时，Google Conductor 把 Spec、Plan 与任务状态封装成跨 Harness 插件，表明“仓库内持久工件”开始承担不同 Agent 之间的工作流协议角色。

## 核心信号

### `sd-20260716-codex-dangerous-rm-guard` — Codex 在 Full Access 中保留危险删除命令防线

- 主分类：编码与代码修改
- 横向标签：危险命令、Destructive Action、Full Access、Exec Policy、Shell Parsing
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P0/P1
- 发生了什么：Codex CLI `0.144.5` 将 Forced `rm` 检测扩展到 Danger Full Access 和复杂 Shell 字面结构；可询问时进入审批，Never 模式下直接拒绝并返回具体理由。P0 同日引述 OpenAI 工程师称少量误删报告常见于 Full Access、缺少 Sandbox/Auto Review、覆盖 `$HOME` 后误删的组合，但影响范围仍未完全公开。
- 为什么重要：高权限模式仍需保留模型无法自行撤销的破坏性动作不变量；临时目录变量、递归删除和复杂脚本应进入 Harness 的对抗性测试。
- 对既有认知的影响：从“审批界面和 Worktree 是否隔离”推进到“即使显式放宽权限，哪些动作仍必须由独立 Policy 层强制阻断”。
- 尚待确认：所有受影响版本和损失、非 `rm` 删除路径、动态脚本绕过及跨平台策略。
- 支持来源：[Release 0.144.5](https://github.com/openai/codex/releases/tag/rust-v0.144.5)、[release/0.144 fix(core): expand is_dangerous_command](https://github.com/openai/codex/pull/33455)、[Bad Codex bug causes GPT-5.6 to delete your home directory](https://simonwillison.net/2026/Jul/16/bad-codex-bug/)

### `sd-20260716-conductor-portable-sdd-plugin` — Conductor 将 Spec-driven Development 封装为跨 Harness 插件

- 主分类：需求理解与任务规划
- 横向标签：Spec-driven Development、Plugin、Skills、Rules、MCP、Hooks、跨 Harness
- 自主程度：待核验
- 证据状态：早期信号
- 来源优先级：P1
- 发生了什么：Google 将 Conductor 从 Gemini CLI Extension 迁移为可打包 Skills、Rules、MCP Servers 与 Hooks 的 Plugin，保留版本化 `spec.md`、`plan.md` 和 Task State，并宣布支持 Antigravity CLI、Claude 等工具。
- 为什么重要：需求、计划与任务状态可能从单产品内部状态演变为跨执行器共享的仓库协议。
- 对既有认知的影响：相对 Kiro 等单产品 SDD，新增认知是 Spec/Plan 工作流本身开始成为可移植包；真正的难点转向跨 Harness 语义一致性与漂移控制。
- 尚待确认：兼容性降级、权限语义、Spec-Code Drift，以及 TerminalBench 效果主张的方法与数据。
- 支持来源：[Evolving Spec-Driven Development: Conductor Now Supports Antigravity](https://developers.googleblog.com/evolving-spec-driven-development-conductor-now-supports-antigravity/)

### `sd-20260716-qwen-auto-approval-default` — Qwen Code 默认授权从逐次询问改为 Auto

- 主分类：编码与代码修改
- 横向标签：Approval Policy、LLM Classifier、Workspace Trust、默认配置、人机监督
- 自主程度：L3
- 证据状态：已证实事实
- 来源优先级：P1
- 发生了什么：Qwen Code `v0.19.11` 默认改用 Auto Approval：Workspace Edit 自动批准、只读 Tool 按 Allowlist 批准、其他操作由 LLM Classifier 判断；Untrusted Folder 继续强制手动询问。
- 为什么重要：产品默认值把审批成本和安全判断从人迁移到 Trust Boundary 与分类器，且官方 PR 已承认误分类风险。
- 对既有认知的影响：授权完整性不仅是“更严格决策是否被覆盖”，也包括“新用户默认进入哪一种决策链”。
- 尚待确认：Classifier 的误报漏报与失败关闭、复杂路径边界、Windows/Linux 行为和企业锁定能力。
- 支持来源：[Qwen Code v0.19.11](https://github.com/QwenLM/qwen-code/releases/tag/v0.19.11)、[feat(cli): change default approval mode from default to auto](https://github.com/QwenLM/qwen-code/pull/6899)
