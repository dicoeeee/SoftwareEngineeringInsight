### 信号

Sentry Seer 在 2025 年 6 月进入 GA 后，可按项目自动扫描 Issue，并以最低 actionability score 决定是否继续自动处理；用户可把 stopping point 设为 root cause、solution、code changes 或 open PR。Seer 会按需读取 stack trace、environment、span、commit、log 等上下文，形成根因、修复方案和代码变更，必要时自动打开 PR。当前官方 API 仍把该流程定义为异步任务，并允许分别请求 `root_cause`、`solution`、`code_changes`、`pr_iteration`、`open_pr` 或 `coding_agent_handoff`；调用需要 `event:write` 或 `event:admin` scope。`repo_name` 可选，省略时可能在所有相关仓库创建 PR，因此仓库目标本身是必须显式治理的爆炸半径。官方所称 38,000 个 Issue、94.5% 根因准确率和节省两年工程时间均属于厂商统计，本记录不把它们视为独立证实效果。

### 为什么重要

分析判断：这条历史补录说明“遥测—根因—代码变更—PR”的闭环在 2025 年已进入付费产品 GA，行业讨论不应只从 2026 年的新发布起算。更值得关注的不是自动开 PR 本身，而是 actionability 阈值、stopping point、API scope 与 repo target 共同构成了可配置控制面；其中任一项默认过宽，都可能把诊断辅助放大为跨仓库写操作风险。

### 尚待确认

- actionability score 的训练数据、校准方式、误触发率和不同项目类型下的稳定性。
- 自动生成 PR 的编译、测试、回归、人工接受率与回滚表现；官方效果指标缺少独立验证。
- 项目级 automation 的默认 stopping point、审批人、审计记录和权限继承方式。
- 省略 `repo_name` 时“所有相关仓库”的判定规则、组织级限制和最小权限建议。
