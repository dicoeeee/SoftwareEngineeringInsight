### 信号

Harness 发布统一 CLI 3.0 Public Beta，以单一二进制、认证路径和 `harness <verb> <noun> [identifier] [flags]` 闭合语法覆盖 pipelines、CD、code、artifacts、IaCM、feature flags、governance 与 audit。CLI 可用 `harness list noun --format json` 枚举资源和动作图，所有命令提供结构化输出与确定性退出码；流水线触发、日志、审批、终止、制品操作和 firewall scan 复用同一语法，并沿用平台 RBAC 与 audit trail。官方仓库以 Apache-2.0 开源；macOS/Linux 已支持，Windows 与部分包管理器仍在 roadmap。

### 为什么重要

分析判断：既有 Harness Worker Agents 证明 reasoning agent 可作为受治理的 pipeline step 运行；新 CLI 把同一交付控制面扩展给平台外部的 coding/runbook Agent。机器可枚举的动作面、稳定输出和确定性结果可降低工具调用歧义，同时复用既有权限与审计边界。CLI 本身不进行 Agent 式规划，因此不能把工具接口直接等同于 L3 Agent；外部 Agent 的真实自主程度仍待核验。

- 外部 Agent 如何获得最小权限、短期身份，以及审批动作能否被策略强制限制为人类主体。
- 厂商关于 Agent 命令占比、效率和上下文节省的主张缺少独立验证，未并入信号事实。

### 信号

JFrog Artifactory 新增 Beta 的 `Agent Packages` repository type，用于存储 skills、plugins、prompts、hooks、MCP server definitions、instructions 与 agents。APM 通过 `apm.yml` 声明依赖，并以 `apm.lock.yaml` 和内容哈希锁定解析结果；组织可在 `.github/apm-policy.yml` 设置 `enforcement: block`、`fetch_failure: block`、指定 registry 和 `allow_non_registry: false`，阻止消费未批准来源。官方示例将固定版本 package 交给 GitHub Agentic Workflows，在人工 approval、sandbox 和 `max-ai-credits` 上限内执行。当前仅支持 local/federated repository 和严格布局的 ZIP；remote、virtual、smart repository、replication 与 TAR.GZ 尚不支持。

### 为什么重要

分析判断：Agent 的行为依赖开始具备传统软件包的版本、锁定、依赖解析、私有 registry 和来源策略，软件供应链控制面由代码与制品扩展到 prompts、skills、hooks 和 MCP definitions。它与 gh-aw 的运行时沙箱、审批和预算形成互补：JFrog 管理 Agent 上下文本身的发布与消费来源，gh-aw 管理加载后的执行边界。

### 尚待确认

- Beta 对签名、attestation、不可变版本、依赖混淆、撤回和策略继承的完整支持。
- policy 文件的信任根，以及恶意仓库修改策略时的保护方式。
- 官方所述恶意指令、隐藏操纵和危险配置扫描仍是未来计划，不能作为已发布能力；误报、漏报与 GA 时间也未知。
