### 信号

Harness 官方文章描述，AI SRE Runbook 由类型化输入输出的有序步骤组成，步骤可以调用连接系统、设置事故字段、按条件分支或遍历列表，并为失败设置错误路径。Runbook 可固定到事故类型供一键运行，也可与告警规则或事故生命周期事件绑定后自动触发。原生 Harness pipeline step 会检查调用者的 pipeline-execute 权限，可等待 rollback 或 hotfix 流水线结束，并把逐步输入、输出、链接及 running/success/failed 状态写入事故时间线。文章能够确认厂商所描述的行为契约，但不能证明它具备动态规划能力或降低 MTTR。

### 为什么重要

分析判断：运维 Agent 的调查与执行不必共享同一自由度。推理系统可以识别事故上下文，实际副作用则交给具备类型、分支、失败路径、调用者权限和审计记录的确定性 Runbook；这与 Google SecOps 的“动态调查 + 确定性 Playbook”形成机制呼应。Runbook 本身属于 L0 固定自动化，不能因部署在 AI SRE 产品中就提升为 L3 Agent。

### 尚待确认

- pipeline 之外的 feature flag、工单和通信动作是否使用同等粒度的调用者身份、审批与最小权限检查。
- 自动触发 Runbook 的幂等键、重复告警合并、并发执行、超时、取消和补偿语义。
- 输入输出写入事故时间线时的秘密脱敏、保留期、访问控制和审计完整性。
- 厂商关于降低 MTTR 和减少人工步骤的效果主张缺少独立验证。

### 信号

Resolve AI 官方产品文章把 Background Agent 描述为由 schedule、deploy/rollback event 或 Slack message 唤醒的持久运维单元。Task 保存跨重启状态，Skill 编码查询或执行方法，memory、files 与 live context graph 保存服务拓扑、依赖和运行基线。厂商列举部署后回归校验、周期性漂移检查、交接报告和工程问题首响四类任务，并声明任何拟议生产变更都必须经过 human review gate。以上均为厂商对自身产品的描述，尚无独立实现或效果验证。

### 为什么重要

分析判断：生产运维 Agent 的时间模型正在从“告警后启动一次会话”扩展到“长期待命、按事件唤醒并跨运行保留状态”。这使上线后短窗口校验和慢性漂移检查可以共享服务基线与历史上下文，但持久记忆也扩大了陈旧知识、跨租户数据、错误基线和长期权限的治理面。

### 尚待确认

- Task、memory、files 与 live context graph 的存储、纠错、保留、租户隔离和敏感数据边界。
- 部署后基线比较和慢性漂移判断的阈值、误报率、取消语义及模型成本。
- human review gate 覆盖哪些动作、由谁批准，以及是否支持细粒度策略、预览、回滚与审计。
