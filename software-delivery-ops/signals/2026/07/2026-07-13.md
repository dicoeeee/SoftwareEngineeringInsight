### 信号

GitHub 在 `gh-aw` v0.82.8 Pre-release 中加入可选的 gVisor Agent 沙箱运行时、可由共享工作流片段声明的沙箱挂载，以及用于在 PR 评论和 Issue 中披露 AI 作者身份的 `disclosure-header` Safe Output。该版本还把 OAuth 或缺失令牌失败传递到结论 Job，并修正 confused-deputy 检查范围与自托管 ARC/DinD Runner 的相关问题。

`2026-07-15` 更新：v0.82.9 Pre-release 进一步加入 `docker-sbx` 沙箱目标、`private-to-public-flows: allow` 显式跨可见性授权、阻止 `toJSON(secrets)` 的编译检查、未限定 `push` 分支的扇出检查，以及 `pull_request` 向 `pull_request_target` 的权限升级检测；CLI proxy 默认策略也改为阻止公共仓库上下文访问私有仓库。以上是官方 Release 与已合并 PR 可确认的实现事实，不代表这些 Preview 机制已经过独立生产验证。

`2026-07-17` 更新：v0.82.10 将 strict security mode 设为默认，取消自动 `sudo` 与 `--enable-host-access`，需要显式选择；同时引入声明式 engine/frontmatter 与 workflow evaluation，首批 10 个工作流通过 BinEval 运行并把结果写入 conclusion usage artifact 和每日评估报告。v0.82.11 将评估扩到额外 24 个工作流；`close`、`assign_to_agent` 等工具默认要求 `issue-intent`，Safe Outputs client token 与 project token 分离，PR 输出使用 fork-backed safe path，并对调用次数上限快速失败。以上版本仍为 Pre-release。

`2026-07-18` 更新：v0.82.12 修复两处此前未覆盖的安全边界。第一，编译器原先只把 `${{ secrets.* }}` 识别为需从 Agent sandbox 排除的凭据环境变量，经 `${{ needs.JOB.outputs.OUTPUT }}` 传入的临时 token 会完整暴露给容器；新版在 `mcp-scripts`、`engine.env` 与 `agent.env` 中统一生成 `--exclude-env`。第二，external threat detection 重建 EngineConfig 时曾丢失 custom OpenAI/Anthropic base URL 与新增 allowlist，导致自定义环境下出现 false negative；新版合并主 engine env 并扩大 allowlist 传播条件。官方 Release 与已合并 PR 可确认缺口和修复，但版本仍为 Pre-release。

`2026-07-19` 更新：v0.82.13 修复 Safe Outputs 审计计数长期错误为 `0` 的三个实现缺陷：`run_summary.json` 字段命名不一致、双文件 artifact 没有 flatten，以及 cache backfill 只改内存而未重新持久化。它同时把 `set_issue_type`、`set_issue_field` 与 `add_labels` 的 intent metadata 改为默认保留，但默认仍兼容纯字符串；只有显式 `issue-intent: true` 才进入 strict mode，显式 `false` 可退回旧路径。官方 Release 与已合并 PR 可确认缺陷和修复，但不能把“默认保留 metadata”表述为“默认强制结构化意图”。

### 为什么重要

这表明 Agentic CI/CD 的治理机制正从提示词约定下沉到可编译、可复用和可审计的工作流字段：执行隔离、挂载范围、输出通道、身份失败和作者披露可以成为流水线契约的一部分。它与 Harness 将 Worker Agent 纳入 RBAC、OPA、审批和审计的基线信号形成跨产品呼应，但当前不足以据此上升为成熟行业趋势。

`2026-07-17` 分析判断：控制面开始同时覆盖“默认拒绝”的运行权限、面向副作用工具的意图门禁、输出令牌隔离与持续评估；这使安全默认值和行为回归测试成为 Agentic Workflow 的发布契约，而不只是部署建议。

`2026-07-18` 分析判断：两项修复同时说明“声明了沙箱/威胁检测”不等于所有凭据来源和自定义 provider 路径都已被隔离。修复本身增强了控制面，但此前可见的 credential 与 false-negative 路径削弱了对 Pre-release 安全完整性的判断，因此关注度动量记为“降温”。

`2026-07-19` 分析判断：写操作意图元数据覆盖面扩大值得肯定，但审计总数长期为零说明控制面的“可审计”仍依赖数据生成、artifact 解析和持久化链路共同正确。修复没有增加独立生产证据，且 strict enforcement 仍需显式开启，因此维持“早期信号”和“降温”。

### 尚待确认

- v0.82.8 何时进入稳定版，以及 gVisor 与 ARC/DinD 组合在生产环境中的兼容性和成本。
- `disclosure-header` 是否会被组织政策、审计工具或代码评审流程实际采用。
- Safe Outputs、网络白名单和挂载配置能否覆盖提示注入、凭据滥用及跨仓库副作用等主要风险。
- strict mode、`issue-intent` 与 fork-backed PR 在大型真实工作流中的兼容性、误阻断率和绕过面。
- BinEval 的样本、评分准则、失败处置与版本回归阈值是否足以支撑生产门禁。
- v0.82.11 及以前经 job output 传入凭据的实际暴露范围、轮换需求，以及其他表达式来源是否仍可能绕过 `--exclude-env`。
- external threat detection 在自定义 provider、代理和动态 allowlist 组合下的回归覆盖与 fail-closed 行为。
- v0.82.13 修复前 Safe Outputs 审计计数错误影响了哪些报告、时间范围和合规判断，是否需要历史回填或更正。
- `issue-intent` 省略、显式 `true` 与显式 `false` 三种模式在组织级策略中如何被强制，纯字符串兼容路径是否会削弱意图门禁。
