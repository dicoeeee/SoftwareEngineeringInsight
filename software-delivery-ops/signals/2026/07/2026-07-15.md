### 信号

GitHub 于 7 月 10 日发布 Public Preview 的 Agentic Autofix。用户可把一个或多个 code scanning alerts 指派给 Copilot：Agent 会探索相关文件、生成修复、重新运行 CodeQL、必要时迭代，最后提交待人工审查的 draft PR。企业策略可以禁用该能力，运行同时消耗 AI Credits 与 GitHub Actions minutes。7 月 14 日新增的 AI security detections 会在 PR 打开或更新时自动运行，并把带 `AI` 标签的 informational findings 显示在 PR，扩展到 CodeQL 内建分析尚未覆盖的语言和框架；当前官方材料没有说明这些 AI findings 会自动触发 Agentic Autofix，因此仅作为同一 code scanning 入口的补充证据。

### 为什么重要

分析判断：安全扫描正从“发现后给出建议”进入“告警—跨文件修改—确定性扫描器复验—草案 PR”的受控闭环。关键边界仍清晰：修复由人显式指派并以 draft PR 交付，7 月 14 日的 AI findings 也不阻止合并；CodeQL 复验只证明对应告警已关闭，不能证明修复没有引入其他问题。

### 尚待确认

- AI security detections 的误报率、漏报率，以及在 CodeQL 未覆盖语言上的真实增益。
- Agentic Autofix 对复杂跨服务漏洞的修复成功率、回归风险和人工复核成本。
- Public Preview 之后的计费、策略控制和合并准入集成是否变化。

### 信号

OpenSRE 7 月 14–15 日的正式 Release 将后台 RCA 扩展为两条互补路径：一条是 Slack Socket Mode 双向网关，把每个 thread 绑定为独立会话，使用必填用户 allowlist、工作区隔离和只记录内容哈希的审计日志；另一条是 API 与 investigation worker 分离的异步队列，使用 PostgreSQL `FOR UPDATE SKIP LOCKED` 领取任务，并可把报告上传至 S3。Terraform 部署将 API/worker 与仅出站的 Slack gateway 分离到 ECS Fargate，秘密来自 SSM SecureString、制品桶权限按任务角色限定。随后版本又加入 Slack 读/回复/成员查询工具，以及 Telegram 后台 RCA 完成通知；通知失败不会反向把已完成调查标记为失败，并对令牌泄漏和消息长度预算做了专门防护。

`2026-07-16` 更新：0.1.2026.7.16 在 Slack turn 抛错时把占位消息替换为可见错误，并仅在内部日志保留异常细节；每个 turn 增加默认 240 秒 watchdog，但阻塞 handler 不能被强制取消，迟到结果仍可能覆盖超时提示。Chat surface 同时隐藏 `slack_send_message`、`telegram_send_message` 等竞争回复工具，并按当前 messaging platform 隔离另一渠道；部署还增加 AWS account guard。

`2026-07-17` 更新：0.1.2026.7.17 为 Slack gateway 增加 liveness heartbeat，并把容器改为非 root 运行；审计工作区迁移到系统临时目录下的 `opensre/workspace`，同时移除旧 Slack backend 基础设施。官方 Release 可确认这些运行加固，但仍未提供生产可靠性数据。

`2026-07-19` 更新：0.1.2026.7.19 把 Rocket.Chat 加入 Agent 可调用消息、后台 RCA 通知与 cron 投递。`rocketchat_send_message` 明确要求 approval，只在 investigation/action surface 可见并从 gateway chat surface 排除；凭据在执行阶段才解析，优先读取环境变量再回退 OS keyring。目的地不满足显式 channel 要求时 fail closed，通知错误会脱敏 token/webhook；同批修复还覆盖 Telegram URL path 中 token 可能出现在验证错误的问题。以上是官方 Release 与已合并 PR 可确认的实现，不代表 Slack、Telegram、GitHub 等其他写工具共享同一审批策略。

`2026-07-23` 更新：OpenSRE 0.1.2026.7.22–7.23 将长期运行 SRE Agent 的接入控制扩展到输入、凭据与连接器配置。7.22 的 Slack gateway 对日志、CSV、JSON、YAML 和图片附件执行流式下载、文件与总字符上限、明显秘密清理及单次视觉描述，并可从组织级 Webapp integration vault 获取连接凭据；共享配置写入器把敏感键和非敏感键强制分流。7.23 又把 PagerDuty、incident.io、Jenkins、Sentry、Datadog、数据库等连接器迁移到同一声明式 setup flow，要求 verify-before-persist，并统一 store、keyring 与 env 的往返；关联 PR 披露旧入口曾使 PagerDuty 等凭据只写 store、部署阶段却检查 env，从而产生配置漂移。参数化测试覆盖失败验证不落盘、默认/可选字段和环境往返，但项目仍为 Public Alpha。

### 为什么重要

分析判断：这组变化的价值不在新增聊天入口，而在把事故调查 Agent 当作长期运行服务处理——队列领取、会话/租户隔离、秘密注入、审计、报告制品和通知失败域都有明确边界。它为“协调器 + 专业 Agent”基线补充了可落地的运行控制，但尚不足以证明 Public Alpha 已具备生产可靠性。

`2026-07-16` 分析判断：协作面还需要错误对用户可见、对外脱敏、回复工具按渠道隔离，并把“超时”与“可取消”分开建模；OpenSRE 明确披露的迟到结果覆盖风险是重要限制。

`2026-07-17` 分析判断：ChatOps gateway 的健康检查、非 root 身份与明确临时工作区补齐了服务运行的基础约束，但它们是加固增量，不改变 Public Alpha 与 L1 辅助洞察定位。

`2026-07-19` 分析判断：第三个消息渠道本身不足以形成新信号；真正有效的增量是把副作用审批、surface scope、目的地失败语义和凭据脱敏写进工具实现。它部分回答了既有消息写操作的治理问题，但审批语义仍是逐工具的，不能推广到无单次审批的 `github_cli`。

`2026-07-23` 分析判断：ChatOps 服务的可靠性不仅取决于队列和工具权限，还取决于所有配置入口是否共享同一验证与持久化事务、所有附件是否经过预算和净化。此次更新补齐了可复用的配置一致性机制与测试证据，但基于变量名的敏感性分类和明显秘密清理仍不能替代完整的数据防泄漏与凭据生命周期治理。

### 尚待确认

- 多实例 worker 在并发、重试、幂等和长时间调查下的可靠性与恢复行为。
- Slack 具备回复、加 reaction、加入频道等写工具后，组织如何配置审批、权限和爆炸半径。
- Public Alpha 的安全评审、真实事故效果、误诊率和运营成本。
- Watchdog 无法取消阻塞 handler，迟到结果可能覆盖超时提示；幂等、状态竞争和恢复语义仍待验证。
- liveness heartbeat 的故障判定、重启/恢复语义，以及临时审计工作区的保留、清理和取证边界。
- Rocket.Chat approval 由谁触发和批准、是否支持组织级强制策略，以及批准记录如何进入审计链。
- Slack、Telegram、Rocket.Chat 与 GitHub 工具的 approval/surface 配置是否有统一清单、默认值与回归测试。
- 未迁移连接器的分支字段、either/or 约束和撤销语义；凭据轮换、并发写入、跨租户 vault 授权及附件秘密清理的误报/漏报仍待验证。
