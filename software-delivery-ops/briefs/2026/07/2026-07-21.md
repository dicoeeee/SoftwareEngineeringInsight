#### 核心判断

GitHub 在 `gh-aw` v0.82.14 中把 64 条 Agentic Workflows 的 evals、按需与每日安全扫描、Safe Outputs 意图提示和 schema fail-fast 汇入正式 Release。Agentic CI 的发布门禁由单一构建检查扩展到行为回归、安全暴露和副作用输出契约，可降低工作流能运行却在质量或权限边界上退化的风险。

#### 变化与技术机制

v0.82.14 的主线不是单一功能，而是把此前分散的评测、安全扫描和 Safe Outputs 控制集中成发布级质量边界，并将 rootless 安装及若干可靠性变化纳入正式版本。工作流先接受 schema 与安全规则编译检查，再由 evals 验证运行结果；DeepSec 和 VulnHunter 以按需或定时工作流持续扫描，最终写操作通过 Safe Outputs 的模式化意图提示进入受控输出通道。

- 64 条 Agentic Workflows 启用 evals，扩大工作流结果的回归检测覆盖。
- DeepSec 与 VulnHunter 以按需和每日工作流执行持续漏洞扫描。
- mutation tools 按运行模式给出 intent guidance（意图指导），编译器对 schema 变化后未重建的工作流 fail fast（快速失败）。

#### 关键解读

##### 行为评测开始成为 Agentic Workflow 的发布资产

评测覆盖从少量示例扩大到 64 条工作流，说明任务结果本身正在被纳入持续回归检查。真正形成门禁仍取决于数据代表性、评分阈值以及失败后是否阻断发布，Release 没有披露这些参数。

##### 安全扫描与副作用输出需要独立治理

DeepSec、VulnHunter 负责发现暴露面，Safe Outputs 负责约束外部写入，两者解决的是不同阶段的风险。团队应分别定义扫描权限、误报处置和 mutation intent 的允许范围，不能以某一类控制替代另一类。

##### 正式发布确认了控制面仍包含运行时可靠性

版本同时处理慢 Runner 的 gateway timeout、terminal safe-output 终态误判和 rootless 安装，说明发布门禁不仅检查模型答案，也必须覆盖工具启动、容器身份和终态传播。此类边界若未被评测覆盖，仍可能把已交付结果误判为失败或触发破坏性重试。

### Harness 用类型化 Runbook 将事故处置动作收敛到权限与时间线

- 原始标题：[How to Build Runbooks That Work — and Automate Them with Harness AI SRE](https://www.harness.io/blog/how-to-build-runbooks-that-work----and-automate-them-with-harness-ai-sre)
- 来源类型：Article
- 来源机构：Harness
- 作者：Ryan Taylor
- 发布时间：2026-07-15

#### 核心判断

Harness AI SRE 把事故 Runbook 实现为带类型化输入输出、条件分支、循环和错误路径的有序步骤链，并可在权限检查后调用 rollback 或 hotfix 流水线。调查推理与副作用执行因此可以采用不同自主层级，实际变更被限制在预定义动作、调用者权限和可追踪状态内。

#### 变化与技术机制

Harness 官方文章将 Runbook 从静态说明文档扩展为事故期间可执行的确定性工作流，同时保留人工一键运行、告警自动触发和生命周期事件触发三种入口。事故类型、告警规则或生命周期事件先选择并触发 Runbook，随后有序步骤读取类型化输入，按条件或列表执行动作；需要变更时调用受权限约束的 Harness pipeline，最后把输出和状态写回事故时间线。

步骤可以调用连接系统、设置事故字段，并为失败设置显式错误路径。原生 pipeline step 会在执行 rollback 或 hotfix 前检查调用者的 pipeline-execute 权限，可等待流水线进入终态；每步输入、输出、链接及 running、success、failed 状态都会写入事故时间线，为审计和 Runbook 调整留下依据。

#### 关键解读

##### 推理自由度不必传递给生产执行器

Runbook 本身属于 L0 固定自动化，Agent 可以负责识别上下文和选择候选处置，但副作用动作仍由确定性步骤执行。这样的分层把“是否理解事故”和“是否允许改变生产”拆成两个可独立验证的问题。

##### 调用者权限必须在执行时重新计算

pipeline step 检查实际调用者的 execute 权限，而不是仅依赖创建 Runbook 时的配置。对自动触发入口还需要补充服务身份、审批和最小权限策略，否则同一 Runbook 在人工与自动场景下可能拥有不同爆炸半径。

##### 时间线记录需要覆盖失败与敏感数据边界

逐步状态写回事故时间线能让失败不再只存在于外部流水线，并为复盘提供证据。与此同时，输入输出可能包含秘密或生产数据，必须明确脱敏、访问控制、保留期和审计完整性。

### Resolve AI 用持久后台 Agent 把上线后校验扩展到连续运维

- 原始标题：[Proactively run production tasks with background agents](https://resolve.ai/blog/proactively-run-prod-with-background-agents)
- 来源类型：Article
- 来源机构：Resolve AI
- 作者：Varun Krovvidi
- 发布时间：2026-07-17

#### 核心判断

Resolve AI 将 Background Agent 设计为由 schedule、部署事件或消息唤醒的持久运维单元，并以 Task、Skill、memory、files 和 live context graph 跨运行保留状态。上线后短窗口回归与跨周慢性漂移由此可以共享服务基线和历史上下文，但持续状态、长期权限及错误记忆也成为新的可靠性与治理边界。

#### 变化与技术机制

该产品主张把一次性告警调查扩展为长期待命的生产任务，覆盖部署后校验、周期性健康检查、值班交接和工程问题首响。时间、deploy/rollback event 或 Slack message 先唤醒 Background Agent，持久 Task 再调用一个或多个 Skills，从 live context graph 读取服务拓扑、依赖与运行基线，随后执行部署校验、漂移检查或报告生成。

Task 保存跨重启状态，Skill 编码查询或动作方法，memory 与 files 在重启后继续存在。Agent 可以自动发起调查和生成报告，但公开文章声明任何拟议生产变更仍需进入 human review gate（人类审阅门）；文章没有给出该门禁的策略、身份和审计实现。

#### 关键解读

##### 运维 Agent 的时间模型从会话转向长期任务

由时间、部署事件或消息唤醒的持久任务，使部署后短窗口校验、跨周漂移检查和交接报告能够复用同一上下文。其价值不只是减少重复查询，而是把任务状态、基线和历史结论变成可持续演进的运行资产。

##### 持久记忆会把错误基线跨运行放大

Task、memory、files 与 live context graph 提升了连续性，也扩大了陈旧知识、跨租户数据和错误服务归属的影响范围。没有版本、来源、新鲜度、纠错和清理机制时，一次错误判断可能持续影响后续检查。

##### 人工审阅门需要落到动作与身份策略

厂商声明生产变更需人工审阅，但没有说明哪些动作进入门禁、由谁批准以及是否支持预览、回滚和细粒度策略。仅有 human-in-the-loop 表述不足以证明高风险动作已经被工程控制约束。
