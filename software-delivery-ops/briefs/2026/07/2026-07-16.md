# 2026-07-16 软件交付与运维 Agent 专题简报

## 今日判断

软件交付 Agent 的治理面正在同时向“工具调用入口”和“上下文供应链”扩展：Harness CLI 3.0 以可枚举语法、稳定输出和平台 RBAC/audit 为外部 Agent 提供确定性执行接口，JFrog 则把 prompts、skills、hooks 与 MCP definitions 变成可版本化、可锁定来源的制品。既有信号的新证据进一步表明，受控执行需要把触发者身份、平台权限、OPA 与工具 allowlist 取交集，同时把超时、错误脱敏和协作渠道隔离作为显式失败域；两项新能力仍处于 Public Beta/Beta，尚无独立生产效果证据。

## 核心信号

### `sdo-20260715-harness-cli-agent-contract` — Harness CLI 3.0 将交付平台暴露为面向 Agent 的确定性工具契约

- 主分类：流水线编排与调度
- 横向标签：Harness、Agent Tool、CLI、RBAC、审计、结构化输出、确定性退出码、制品、IaC、OPA
- 自主程度：待核验（CLI 本身不是 Agent）
- 证据状态：早期信号
- 来源优先级：P1
- 发生了什么：Harness CLI 3.0 Public Beta 用统一二进制、认证路径和闭合 verb/noun 语法覆盖流水线、部署、审批、制品、IaC、策略与审计，并提供机器可枚举动作图、结构化输出和确定性退出码。
- 为什么重要：外部 coding/runbook Agent 可以通过稳定机器契约调用交付平台，同时复用已有 RBAC 与 audit trail；它不是又一个聊天入口。
- 对既有认知的影响：把 Harness 的 Agent 治理从平台内部 Worker Agent step 扩展到平台外部 Agent 工具调用，但不能把 CLI 本身夸大为 Agent。
- 尚待确认：Beta 兼容性、幂等与并发语义、破坏性动作确认、最小权限/短期身份，以及审批能否强制保留人类主体。
- 支持来源：[Announcing the Harness CLI: Built for Humans and Agents](https://www.harness.io/blog/announcing-harness-cli)、[harness/cli](https://github.com/harness/cli)

### `sdo-20260714-jfrog-agent-packages` — JFrog 将 Agent primitives 纳入可版本化、可强制来源的制品治理

- 主分类：构建、缓存与制品管理
- 横向标签：JFrog、Artifactory、APM、Agent Supply Chain、Skills、Prompts、Hooks、MCP、Policy-as-Code、版本锁定、来源强制
- 自主程度：L3 受控执行（针对官方 gh-aw 示例；仓库本身为治理控制面）
- 证据状态：早期信号
- 来源优先级：P1
- 发生了什么：Artifactory 的 Beta `Agent Packages` repository type 可存储 Agent primitives；APM 用 manifest、锁文件和哈希固定依赖，组织策略可阻止非指定 registry，官方示例再叠加 gh-aw 的 approval、sandbox 和 AI credits 上限。
- 为什么重要：影响 Agent 行为和工具调用的上下文开始像软件包一样具备版本、依赖、可信来源和消费策略。
- 对既有认知的影响：在 gh-aw 运行时治理之外补充“加载前”的 Agent 上下文供应链控制；二者是互补层，不是重复信号。
- 尚待确认：签名/attestation、不可变版本、依赖混淆、撤回、策略信任根与 GA；恶意内容扫描仍是 roadmap，不能当作已发布事实。
- 支持来源：[Managing AI Agent Primitives Like Real Software Packages with APM and JFrog](https://jfrog.com/blog/managing-ai-agent-primitives-with-apm-and-jfrog/)、[Agent Packages Repositories](https://docs.jfrog.com/artifactory/docs/agent-packages-repositories)
