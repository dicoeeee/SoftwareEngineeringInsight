### monday.com 用远程 Sandbox、规则门禁与置信度路由约束 Agent PR 自动合并

- 原始标题：[AI Teammates: How monday.com runs production AI agents on Amazon Bedrock](https://aws.amazon.com/blogs/machine-learning/ai-teammates-how-monday-com-runs-production-ai-agents-on-amazon-bedrock/)
- 来源类型：Article
- 来源机构：AWS 与 monday.com
- 作者：Claudio Mazzoni、Erez Drutin、Moran Zilberstein、Netanel Abergel、Ofek Dayan
- 发布时间：2026-07-22

#### 核心判断

monday.com 将 Agent PR 的交付路径设计为远程 Sandbox（隔离的远程测试环境）自动部署、测试与生产流量回放，再经过 PR Guardrails（将工程规范编码为自动检查的 PR 门禁）和置信度评估后决定自动合并或转人工。该路径把 Agent 的代码生成、隔离验证、工程规范、回滚表现和发布权限串成同一准入链，减少把单一模型结果直接提升为生产变更的风险。

#### 变化与技术机制

文章描述的事件入口来自 Slack、monday 工作项和 GitHub PR；SNS 按主题分发到每团队 SQS 队列，EKS 消费者将任务分派给对应的 Agent runner pod（承载单个 Agent 会话的 Kubernetes 容器实例）。实时任务状态、锁、心跳和消息日志使用 ElastiCache，会话工作区与记忆使用 EFS，持久 transcript（执行过程记录）、快照、制品与评测保存到 S3，使一次会话可在不同 pod 间恢复。

对 PR 的关键控制位于提交之后、人工评审之前：每个 PR 自动部署到专属远程 Sandbox，运行测试、检查和生产流量回放；PR Guardrails 将指标、特性开关、可观测、数据与微服务规范等转为自动检查，部分规则可阻断。自动合并还综合阻断规则结果、当前 Agent 版本的评测轨迹、同类变更历史回滚率和 Sandbox 结果；不足阈值时交给人工。

#### 关键解读

##### 自动合并需要把“证据组合”而非模型置信度作为开关

模型给出的自我评价无法替代隔离环境、确定性规则和变更历史。将规则结果、版本化评测、变更类别和回滚记录合并，能让合并阈值随风险证据调整；前提是每项信号可追溯、可校准且高风险缺口默认转人工。

##### 会话可恢复性与交付证据必须分层保存

短生命周期 runner 需要低延迟状态来处理锁和心跳，也需要共享工作区以支持恢复；但审计、制品与评测应进入持久存储。将三类数据混放会同时放大成本、恢复复杂度或证据缺失风险。

### KernelDiag 以结构化 Evidence Graph 连接内核崩溃制品与方法级根因定位

- 原始标题：[KernelDiag: Agent-Based Root Cause Diagnosis for Kernel Crashes](https://arxiv.org/abs/2607.17722)
- 来源类型：Article
- 来源机构：Tianjin University
- 作者：Weijing Wang、Zan Wang、Dong Wang、Haichi Wang、Junjie Chen
- 发布时间：2026-07-20

#### 核心判断

KernelDiag 提出将日志、syscall、crash report、源代码与特定内核配置先映射并交给制品专用 Agent，再把逐步推断的因果依赖汇入 Evidence Graph（证据图），以定位故障方法并生成可行动解释。它将根因诊断从对单一日志或单次模型回答的依赖，转为能够检查证据如何汇合、为何指向具体代码位置的结构化过程。

#### 变化与技术机制

Linux kernel 的 crash 诊断同时依赖低层日志、调用信息、crash report、代码语义和特定配置；这些制品格式不同，故障传播又未必沿线性调用链展开。KernelDiag 首先进行 log-to-code mapping（日志到代码映射：把日志中的函数、位置或调用线索关联到源码），再让面向不同制品的 Agent 迭代推理源级程序语义和 crash 条件，逐步组织出含依赖关系的 Evidence Graph。

论文将该框架用于 KGYM 的 279 个真实 Linux kernel crash（覆盖 kernel 4.x 至 6.x、72 个子系统），在文件和方法两个粒度与 LinuxFL+、Agentless 比较。Top@k 指标表示正确故障文件或方法是否出现在模型给出的前 k 个候选中；论文报告无显式提示的困难设置下定位有所提升，解释质量也由人工和 LLM 辅助方式评价，但这些结果仍限定于其基准、模型、实现和评估设计。

#### 关键解读

##### RCA（根因分析）的关键产物应是可审查的证据结构，而非单一结论文本

将诊断制品和推断链写入图结构，能够让操作员检查各节点来自何处、哪些路径支撑定位，以及何处需要补充证据。这比只输出“可能原因”更适合接入人工复核、变更审批和事故复盘。

##### 方法级定位不等于可以自动修复

即使定位准确，内核故障的修复仍涉及补丁可行性、回归范围、环境差异和发布风险。诊断 Agent 可以先作为只读调查层，只有经过独立测试、变更审阅和回滚计划后，才应连接受控执行路径。

**适用条件。** 适合日志、trace、配置、代码和复现步骤可关联，且团队需要在高复杂度故障中保留诊断依据的内核、基础设施或平台运维场景。

**不宜照搬。** 不能把基准中的定位表现直接视为生产自动修复能力；缺少稳定代码映射、可信配置快照、专家复核和回滚通道时，Evidence Graph 只能作为调查辅助。
