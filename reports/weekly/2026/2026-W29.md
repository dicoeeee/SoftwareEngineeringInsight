## 本周核心判断

- **Agent 控制面的竞争点从“是否有 Sandbox/Approval”转向控制链能否保持完整。** 开发侧连续出现 Worktree 越界、审批呈现与 Shell 解析、Session 恢复、MCP 信任基线等具体失效面；交付与运维侧则把身份、动态授权、OPA、工具 Allowlist、Safe Outputs 和紧急停止组合成服务端控制面。可确认的是产品机制与修复，不能据此直接证明安全效果。

## 趋势与关联

1. **授权正在成为跨模式、跨进程、跨资产的状态机，而非一次弹窗。** `sd-20260715-claude-approval-integrity`、`sd-20260714-goose-execution-guardrails` 和 `sd-20260714-copilot-mcp-trust-baseline` 分别覆盖策略合并、上下文压缩/取消和资产变更；`sdo-20260630-harness-worker-agents` 与 `sdo-20260528-google-sre-actuation-control-plane` 再把身份、动态降权、确定性执行和紧急停止加入服务端。相对基线中“RBAC、OPA、审批门约束 Agent”的概括，本周新增的是一组可直接测试的不变量与失败模式。
2. **多 Agent 的核心对象从数量变成有状态任务图。** `sd-20260718-qwen-background-delegation-guardrails` 与 `sd-20260715-opencode-subagent-depth-limit` 将后台默认、写入边界、模型限流和递归深度参数化；`sd-20260715-claude-background-agent-lifecycle` 与 `sdo-20260713-opensre-async-chatops` 证明 Kill、Resume、Completion、Watchdog 和迟到结果仍会漂移。它延伸了基线的“异步、多 Agent、Durable Execution”，但显示 Durable 不能只等同于持久 Session。
3. **Agent 上下文正同时成为可移植协议与供应链攻击面。** `sd-20260716-conductor-portable-sdd-plugin`、`sd-20260430-kiro-spec-driven` 与 `sd-20260717-qwen-channel-memory-controls` 把 Spec、Hooks、Task State 和 Memory 持久化；`sdo-20260714-jfrog-agent-packages` 与 `sdo-20260602-github-review-actions` 则显示来源锁定、Head-branch Instructions 和 Runner 配置会改变执行边界。关联逻辑是：上下文越能跨 Harness、Workspace 和协作入口复用，就越需要版本、所有者、审批、签名和审计。
4. **从诊断到变更的闭环正在按风险分层，而非统一走向全自动。** `sdo-20260714-grafana-cross-source-assistant` 与 `sdo-20260717-datadog-cloud-cost-skill` 保持 L1 调查；`sdo-20260710-github-agentic-autofix`、`sdo-20260416-gitlab-agent-control-plane` 和 `sdo-20250617-sentry-seer-telemetry-autofix` 分别以 Draft PR、MR 准入或 Stopping Point 控制 L2/L3 副作用。相对基线中的早期“生产遥测闭环到代码修复”，历史补录说明能力出现更早，但独立效果证据仍没有同步升级。
5. **产品默认值出现方向相反的证据。** `sd-20260716-codex-dangerous-rm-guard`、`sd-20260715-claude-approval-integrity` 和 `sdo-20260711-gh-aw-sandbox-disclosure` 强化失败关闭与默认严格；`sd-20260716-qwen-auto-approval-default` 则把 Workspace 内写入默认交给自动判断，`sdo-20260718-opensre-github-chat-writes` 甚至对 GitHub 写操作取消逐次审批。矛盾并不表示某一方向已胜出，而是说明“默认授权模式、不可自动批准的例外和 Token/Repo 边界”应成为产品对比的首要字段。

## 对我们的启示

- **产品机会：** 建立统一的 Agent Control Plane 评测面板，直接比较默认 Approval、不可自动批准的动作、身份/Token Scope、Workspace/Repo Ownership、Network Egress、Audit Correctness、Rollback 和 Kill/Resume 语义，而不是只比较模型与功能数量。
- **产品机会：** 为多 Agent 调度提供任务图可视化、写入所有权、递归深度、按模型容量、交接 Contract 和子结果验证；这些能力已经从最佳实践变成多个 Harness 的显式参数。
- **技术架构：** 高风险系统应分离 Reasoning 与 Deterministic Actuation，让独立 Policy 层负责动态授权、Dry-run、风险分级、短期身份、速率限制、结果验证和全局停止；模型不能同时决定动作、授权和审计结果。
- **技术架构：** Spec、Rules、Hooks、Skills、MCP 与 Memory 应按供应链工件治理：需要来源、版本、Hash/Signature、Owner、作用域、变更后重新审批、原子更新和可追溯回滚。
- **落地条件：** Preview/Beta 能力进入生产前，必须做跨平台、Self-hosted Runner、Symlink/路径、Shell 等价语法、Credential 来源、Custom Provider、并发/恢复和审计数据正确性测试。
- **主要风险：** 厂商自报效果可能把“功能已发布”误写成“闭环有效”；Denylist、Sandbox 标签或存在审计字段都不能替代端到端验证。自动开 PR/合并、任意 API、跨仓库默认目标和长期 Memory 是当前爆炸半径最高的四类入口。
